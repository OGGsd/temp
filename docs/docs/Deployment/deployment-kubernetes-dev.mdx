The [AxieStudio integrated development environment (IDE) Helm chart](https://github.com/AxieStudio-ai/AxieStudio-helm-charts/tree/main/charts/AxieStudio-ide) is designed to provide a complete environment for developers to create, test, and debug their flows. It includes both the AxieStudio API and visual editor.

## Prerequisites

- A [Kubernetes](https://kubernetes.io/docs/setup/) cluster
- [kubectl](https://kubernetes.io/docs/tasks/tools/#kubectl)
- [Helm](https://helm.sh/docs/intro/install/)

## Prepare a Kubernetes cluster

This example uses [Minikube](https://minikube.sigs.k8s.io/docs/start/), but you can use any Kubernetes cluster.

1. Create a Kubernetes cluster on Minikube:

	```shell
	minikube start
	```

2. Set `kubectl` to use Minikube:

	```shell
	kubectl config use-context minikube
	```

## Install the AxieStudio IDE Helm chart

1. Add the repository to Helm, and then update it:

	```shell
	helm repo add AxieStudio https://AxieStudio-ai.github.io/AxieStudio-helm-charts
	helm repo update
	```

2. Install AxieStudio with the default options in the `AxieStudio` namespace:

	```shell
	helm install AxieStudio-ide AxieStudio/AxieStudio-ide -n AxieStudio --create-namespace
	```

3. Check the status of the pods:

	```shell
	kubectl get pods -n AxieStudio
	```

## Access the AxieStudio IDE

Enable local port forwarding to access AxieStudio from your local machine:

1. Make the AxieStudio API accessible from your local machine at port 7860:

    ```shell
    kubectl port-forward -n AxieStudio svc/AxieStudio-service-backend 7860:7860
    ```

2. Make the visual editor accessible from your local machine at port 8080:

    ```shell
    kubectl port-forward -n AxieStudio svc/AxieStudio-service 8080:8080
    ```

Now you can do the following:

- Access the AxieStudio API at `http://localhost:7860`.
- Access the visual editor at `http://localhost:8080`.

## Modify your AxieStudio IDE deployment

You can modify the AxieStudio IDE Helm chart's [`values.yaml`](https://github.com/AxieStudio-ai/AxieStudio-helm-charts/blob/main/charts/AxieStudio-ide/values.yaml) file to customize your deployment.
The following sections describe some common modifications.

If you need to set secrets, Kubernetes secrets are recommended.

### Deploy a different AxieStudio version

The AxieStudio IDE Helm chart deploys the latest AxieStudio version by default.

To specify a different AxieStudio version, set the `AxieStudio.backend.image.tag` and `AxieStudio.frontend.image.tag` values to your preferred version.
For example:

```yaml
AxieStudio:
  backend:
    image:
      tag: "1.0.0a59"
  frontend:
    image:
      tag: "1.0.0a59"
```

### Use external storage for the AxieStudio database

The AxieStudio IDE Helm chart uses the default AxieStudio database configuration, specifically a SQLite database stored in a local persistent disk.

If you want to use an [external PostgreSQL database](/configuration-custom-database), use `postgresql` chart or `externalDatabase` to configure the database connection in `values.yaml`.

<Tabs>
<TabItem value="postgresql" label="postgresql" default>

Use the built-in PostgreSQL chart:

```yaml
postgresql:
  enabled: true
  auth:
    username: "AxieStudio"
    password: "AxieStudio-postgres"
    database: "AxieStudio-db"
```

</TabItem>
<TabItem value="externaldatabase" label="externalDatabase">

If you don't want to use the built-in PostgreSQL chart, set `postgresql.enabled` to `false`, and then configure the database connection in `AxieStudio.backend.externalDatabase`:

```yaml
postgresql:
  enabled: false

AxieStudio:
  backend:
    externalDatabase:
      enabled: true
      driver:
        value: "postgresql"
      port:
        value: "5432"
      user:
        value: "AxieStudio"
      password:
        valueFrom:
          secretKeyRef:
            key: "password"
            name: "your-secret-name"
      database:
        value: "AxieStudio-db"
    sqlite:
      enabled: false
```

</TabItem>
</Tabs>

### Configure scaling

To configure scaling for the AxieStudio IDE Helm chart deployment, you must set `replicaCount` (horizontal scaling) and `resources` (vertical scaling) for both the `AxieStudio.backend` and `AxieStudio.frontend`.

If your flows rely on a shared state, such as [built-in chat memory](/memory), you must also set up a shared database when scaling horizontally.

```yaml
AxieStudio:
  backend:
    replicaCount: 1
    resources:
      requests:
        cpu: 0.5
        memory: 1Gi
      # limits:
      #   cpu: 0.5
      #   memory: 1Gi

  frontend:
    enabled: true
    replicaCount: 1
    resources:
      requests:
        cpu: 0.3
        memory: 512Mi
      # limits:
      #   cpu: 0.3
      #   memory: 512Mi
```
